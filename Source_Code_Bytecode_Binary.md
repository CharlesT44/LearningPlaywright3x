# Source Code vs Bytecode vs Binary Code

> Example file: `Playwright_3X_Batch\01_Chapter_Java_Script\01_HelloWorld.js`
>
> Code: `console.log("Hello World!");`

## Comparison Table

| Aspect | Source Code | Bytecode | Binary Code (Machine Code) |
|---|---|---|---|
| **Definition** | Human-readable instructions written in a programming language | Intermediate representation between source and machine code | Raw instructions the CPU executes directly (0s and 1s) |
| **Readability** | ✅ Readable by humans | ❌ Not human-readable | ❌ Not human-readable |
| **Our Example** | `console.log("Hello World!");` | `1a 00 01 00 02 00 03 00 04 00 05 00 06 00 07 00 08 00 09 00` (V8 Ignition bytecode — hex dump) | `b8 01 00 00 00 48 8d 3c 25 04 20 40 00 e8 0a 00 00 00` (x86-64 — hex dump) |
| **Abstraction Level** | High-level (close to human language) | Medium-level (between high and low) | Low-level (close to hardware) |
| **Platform Dependent** | ❌ No — write once, run anywhere (for interpreted/JIT languages) | ✅ Yes — often engine/runtime-specific | ✅ Yes — tied to specific CPU architecture (x86, ARM, etc.) |
| **Execution** | Not executed directly — must be compiled/interpreted | Executed by a virtual machine (VM) or JIT-compiled to machine code | Executed directly by the CPU |
| **Speed** | Slowest (interpretation overhead) | Medium (faster than interpretation, slower than native) | Fastest (direct hardware execution) |
| **Storage Size** | Smallest (plain text) | Medium | Largest |
| **Generation Tool** | Written by developers in a text editor | Generated by compiler/interpreter (e.g., V8 Ignition for JS, JVM `javac` for Java) | Generated by assembler or JIT compiler |
| **Example Languages** | JavaScript, Python, Java, C, C++ | Java bytecode (`.class`), Python bytecode (`.pyc`), V8 Ignition bytecode | Native executables (`.exe`, ELF, Mach-O) |
| **Can you modify it?** | ✅ Yes — edit the text file | ⚠️ Possible but impractical | ❌ Very difficult |

## Visual Flow

```
Source Code                          Bytecode                           Binary Code
"console.log('Hello World!');"  →  [Intermediate IR]               →  [CPU Instructions]
         │                              │                                    │
   Written by you                 Generated by V8 engine            Executed by your CPU
   (HelloWorld.js)                (in memory, not saved to disk)    (in memory)
```

## How It Works for This JavaScript Example

1. **Source Code** — You write `console.log("Hello World!");` in `HelloWorld.js`. This is plain text, easy to read and edit.

2. **Parsing** — Node.js (V8 engine) parses the source into an **Abstract Syntax Tree (AST)**.

3. **Bytecode Generation** — V8's **Ignition interpreter** compiles the AST into **bytecode**. Bytecode is a compact, low-level set of instructions understood by the virtual machine. It is **not saved to a file** — it lives in memory during execution.

4. **JIT Compilation to Binary** — For hot code (executed many times), V8's **TurboFan compiler** compiles the bytecode into **native machine code** (binary) specific to your CPU (x86-64, ARM64, etc.). This binary runs directly on the hardware at maximum speed.

5. **Execution** — The CPU executes the binary instructions, and `"Hello World!"` appears in your terminal.

## Key Takeaway

> **Source code** is what you write. **Bytecode** is a middleman the engine uses to start running your code quickly. **Binary code** is the final native form that actually runs on the hardware.
